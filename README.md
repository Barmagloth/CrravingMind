# CravingMind («Жаждущий Разум»)

**Версия спецификации:** 1.6.1 — Релиз-кандидат
**Парадигма:** Вычислительный дарвинизм. Оптимизация через аппаратный дефицит.

---

## Что это

CravingMind — система, в которой LLM-агент эволюционирует под давлением жёсткого ресурсного дефицита. Агент не получает инструкций о том, как стать лучше. Он получает ограниченный токеновый бюджет, набор задач и четыре числа обратной связи по результатам. Всё остальное — его проблема.

**Целевой артефакт** — автономно написанный алгоритм сжатия текста: Python-функция `compress(text, target_ratio) → compressed_text`, которая сокращает произвольный текст с минимальной потерей смысла. Функция не использует LLM-вызовов, работает на чистом Python и может быть встроена в любой пайплайн с нулевой стоимостью инференса.

---

## Ключевые концепции

### Экономика выживания

Агент работает в условиях единого токенового бюджета на эпоху (фиксированный батч задач). Из этого бюджета оплачивается всё: выполнение задач, рефлексия, работа с памятью, переписывание алгоритма. Превышение бюджета — `OOM_KILLED`: мгновенный откат всех изменений.

Система вводит давление постепенно через три фазы:

| Фаза | Эпохи | Что включается |
|------|-------|----------------|
| **Инкубатор** | 1–10 | Hard Cap, Circuit Breaker, базовая телеметрия |
| **Становление** | 11–25 | Долгосрочная память (`bible.md`), R&D-фонд, DIY-роутинг |
| **Полное давление** | 26+ | Режим «Крыса», Протокол Шрамирования, фильтр дубликатов |

### Память агента

С Фазы 2 агент самостоятельно ведёт два файла:

- **`bible.md`** — накопленные правила и эвристики для функции `compress()`. Разрастание файла пожирает бюджет; агент обязан сам его компактифицировать.
- **`graveyard.md`** — история провальных гипотез с LRU/TTL-вытеснением.

Агент пишет Python-скрипты для селективного чтения собственной памяти (DIY Routing). Кривой скрипт в бесконечном цикле убивает агента.

### Внешний Судья

Детерминированный оркестратор без семантического понимания. Проверяет каждый результат через пайплайн:

1. Генерирует 10 вопросов по исходному тексту (один раз, при сборке бенчмарка).
2. Подаёт `compressed_text` эталонной LLM и получает ответы на те же вопросы.
3. Сравнивает ответы по двум осям: **семантика** (cosine similarity эмбеддингов) и **факты** (entity F1).

Агент получает четыре числа: `compression_ratio`, `semantic_score`, `entity_score`, `pass/fail`.

### Бенчмарк и защита от переобучения

Замороженный Parquet-файл, разделённый на рабочий пул (70%) и holdout-пул (30%). Агент не получает обратную связь по holdout-задачам — расхождение рабочего и holdout-скора детектирует переобучение. Раз в N эпох часть пула ротируется.

Первая итерация (MVP) проверяет три типа текста: **дискурс**, **иголки** (факты в длинном тексте), **код**.

## Архитектурные решения

- **Gradient-free оптимизация** через четыре числа обратной связи — намеренное ограничение, стимулирующее генерализацию.
- **Мёртвый эталон** (замороженный бенчмарк) гарантирует воспроизводимость оценки; при смене модели версия эталона инкрементируется.
- **CUSUM-мониторинг** дрейфа `success_rate` — автоматический флаг `DATA_DRIFT` при отклонении > 2σ.
- **Ручная калибровка** прокси-метрик каждые 20 эпох: автоматические метрики не являются истиной.

## Текущий статус

Спецификация v1.6.1 находится в статусе **релиз-кандидат**. Реализация не начата.

Открытые вопросы:
- Устойчивость сходимости на gradient-free оптимизации (решается экспериментально).
- Потенциал переобучения на замороженном бенчмарке при 100+ эпохах (смягчается holdout-ротацией).
- Потолок качества pure Python-компрессора по сравнению с fine-tuned нейросетями (не цель; цель — нулевая стоимость инференса).
