# CravingMind — «Жаждущий Разум»

**Версия спецификации:** 1.6.1 | **Статус:** Релиз-кандидат
**Парадигма:** Вычислительный дарвинизм. Оптимизация через аппаратный дефицит.

---

## Что это

CravingMind — исследовательская система, в которой LLM-агент работает в условиях жёсткого дефицита ресурсов и вынужден оптимизировать собственное поведение, чтобы выжить. Агент не получает инструкций по самосовершенствованию — он получает ограниченный бюджет токенов, набор задач и четыре числа обратной связи. Всё остальное — его проблема.

**Цель проекта — не сам агент (он расходный материал), а артефакт**, который агент создаёт в процессе выживания: Python-функция `compress(text, target_ratio) → compressed_text`, способная сжимать произвольный текст с минимальной потерей смысла без единого LLM-вызова. Функция встраивается в любой пайплайн с нулевой стоимостью инференса.

---

## Ключевые концепции

### Экономика токенов

Единственная валюта существования агента — токены. Жёсткий лимит (`Hard Cap`) на эпоху (фиксированный батч задач). Превышение ведёт к `OOM_KILLED` и полному откату изменений. Бюджет единый для всего: рабочие задачи, рефлексия, чтение памяти, переписывание алгоритма.

**Circuit Breaker:** ни одна задача не имеет права потребить более 15% бюджета эпохи. Перебор → `PARTIAL_FAIL`, эпоха продолжается.

**Скупой Пульс:** единственная бесплатная телеметрия — строка `[B:14050|C:412]` (остаток бюджета | цена прошлого шага) в системном промпте каждого шага. Детализация расходов — только через платный вызов инструмента аудита.

### Три фазы давления

Ограничения включаются последовательно. Агент не уведомляется о смене фазы.

| Фаза | Эпохи | Что добавляется |
|------|-------|-----------------|
| **Инкубатор** | 1–10 | Hard Cap, Circuit Breaker, Скупой Пульс. Памяти нет, штрафов нет. Цель: написать первую рабочую `compress()`. Экспоненциальный венчур — увеличенный стартовый бюджет, испаряющийся к эпохе 5. |
| **Становление** | 11–25 | `bible.md` и `graveyard.md`, DIY Routing, эндогенная компрессия памяти, R&D Slush Fund. |
| **Полное давление** | 26+ | Режим «Крыса» при критическом голодании, Протокол Шрамирования (LRU/TTL), Duplicate Filter. |

### Цифровой капитализм (R&D Slush Fund)

Сэкономленные в конце успешной эпохи токены конвертируются в R&D-бюджет следующей эпохи с убывающей отдачей (потолок 30% базового бюджета). Условие доступа: `success_rate ≥ 50%` и отсутствие `OOM_KILLED`. Провалил эпоху — получаешь голый базовый бюджет без фонда развития.

### Память агента

С Фазы 2 агент самостоятельно ведёт два файла и платит за работу с ними из своего бюджета:

- **`bible.md`** — накопленные правила и эвристики. Если файл разросся и начал пожирать бюджет — агент обязан сам его компактифицировать. Вырежет критически важное — провалит следующие задачи.
- **`graveyard.md`** — история провальных гипотез. Логические ошибки вытесняются по LRU/TTL; технические сбои (`immutable`) хранятся вечно.

**DIY Routing:** агент самостоятельно пишет Python-скрипты для селективного чтения памяти — только релевантные фрагменты, а не весь файл. Скрипт в бесконечном цикле сжигает бюджет и убивает агента.

```
                        ▼
┌─────────────────────────────────────────────────────┐
│              ДАШБОРД ОПЕРАТОРА                       │

│  starvation_rate · oom_count · rnd_earned/spent      │
└─────────────────────────────────────────────────────┘
```

### Судья и оценка качества

Детерминированный внешний процесс. Все LLM-вызовы с `temperature=0` и фиксированным seed. Версия модели прибита к конкретному snapshot.

**Пайплайн оценки задачи:**
1. Судья задаёт 10 замороженных вопросов по `compressed_text`.
2. Каждая пара ответов оценивается по двум осям:
   - **Семантика:** cosine similarity эмбеддингов (сохранение смысла)
   - **Факты:** entity F1 по именованным сущностям (числа, даты, имена)
3. `task_score = 0.5 × semantic_score + 0.5 × entity_score`
4. `PASS if task_score ≥ 0.85`

**Type-specific валидаторы** (агент не знает тип):
- `needle` — требует entity_f1 = 1.0 по каждому вопросу; частичные потери фактов обнуляют score_i
- `code` — только entity_f1 (exact match нормализованных ответов), cosine игнорируется
- `discourse` — только базовый score

**Защита от переобучения:** бенчмарк разделён 70/30. Holdout-пул агент не видит; оператор наблюдает расхождение рабочего и holdout-скора в дашборде. Ротация 10% пула раз в N эпох.

**Ручная калибровка** раз в 20 эпох: оператор вручную проверяет 10 пар `{оригинал, compressed_text}` из holdout. Автоматические метрики — прокси-оценки, а не истина.

### Целевой артефакт `compress.py`

```python
def compress(text: str, target_ratio: float) -> str:
    """
    Сжимает произвольный текст до заданного коэффициента.

    Args:
        text: исходный текст (любой домен, любой язык)
        target_ratio: целевое отношение len(output)/len(input), например 0.3

    Returns:
        сжатый текст
    """
```

Pure Python, без LLM-вызовов, без сетевых запросов. Разрешённые зависимости: stdlib + `numpy`, `scikit-learn`, `spacy`, `nltk`. После каждой успешной эпохи автоматически архивируется с полными метриками качества.

## Текущий статус

Спецификация v1.6.1 — **релиз-кандидат**. Реализация не начата.

**MVP** охватывает три типа текстов: дискурс, иголки, код.

**Планируемые расширения бенчмарка:** структурированные данные (JSON/XML/SQL), мультитёрновые диалоги, инструкции, таблицы, тексты с противоречиями.

**Открытые вопросы:**
- Устойчивость сходимости на gradient-free оптимизации через 4 числа (решается экспериментально)
- Потенциал переобучения на замороженном бенчмарке при 100+ эпохах (смягчается holdout-ротацией)
- Потолок качества pure Python-компрессора — не цель; цель — нулевая стоимость инференса
